Executive Summary

The application we are redesigning the interface for is a command line based employee management system. This original application aligned with the communication basic metaphor and catered to advanced users that were familiar with the system making the system complex to use for the completion of tasks. Our redesign of this system has basic functionality such as hour tracking and user management for all employees along with higher level management functions for system admins which can add new users, teams, and projects along with managers who have unique functions to manage their team such as adding and removing team members, and editing team member work hours such as for cases such as employee absence. Our goal with the design of this system is to create a web-based user interface that is intuitive to use and therefore can be used without much previous instruction which is important in cases where such an interface is rolled out to a company with many employees. To make our interface this intuitive we will be making use of the user’s existing knowledge of similar systems and the outside world to make features visible and at a glance understandable to minimize the gulfs of execution and evaluation. Starting at the login page we will use the user’s existing understanding of how to use a login page by making it similar to most common login pages. From there, the user is directed to a screen with their functionality on the left-hand side. Left side function selection is another common aspect of existing web interfaces so we are making use of the user’s mental models for this as well. This increases the visibility of the function selection since people will be used to looking there to find function selection. This is much more simple than the original design where all the functions are just listed at the top in small cluttered font making functions difficult to read. Since there are three types of users, each with some shared and some unique functions, to allow easy adaption to employee promotions, we made the shared functions all at the top of the left-hand selection, and all in the same order. This way if an employee is promoted to manager or system admin, the functions they are used to will still be in the same spot reducing the need to re-learn the interface. All functions will include a written description of their topic to make it clearly visible what can be done on each tab. The shared functions also include icons so that they can easily be used by employees who may not understand the written function description. The original system responded with mostly system descriptive errors and the system would stop when an error occurred causing confusion among users who were not familiar with the inner workings of the system requiring advanced knowledge for simple mistakes. The redesign returns better feedback for errors and the results of system actions to help bridge the gulf of evaluation. For the navigation bar, will make use of the user’s familiarity with real world objects as metaphors to help with navigation using the icons such as a magnifying glass for searching for employees which can be used to search for something in the physical world and in many interfaces symbolizing a search function. We also made use of combining similar functions under one tab to avoid over-cluttering the navigation tab reducing the likelihood of accidental clicks on the wrong tab such as by combining all team management features of the manager under a team management tab instead of having one tab for each feature. Within the features, we also make use of the user’s frames along with the other design principles such as affordances and transfer to make it clear what results will occur from the user’s actions. For example in the managers UI under the team management tab which combines the add employee to team, remove an employee from the team, and set team project functions, each function has hints to make results of actions more clear to the user. To start off, knowing the user just selected the tab on the left side, we set the less destructive functions closer to the left side which is to add employees. This should reduce any accidental removals of team members since the user would have to be more intentional about going to the other side of the screen for that functionality. For all inputs, they will highlight red while being filled out until they are in the properly formatted structure. This highlighting will provide some transference to the user with how stop lights work, with green meaning you are good to go or nothing has changed and red meaning stop and don’t proceed with the change indicating something isn’t valid. The original design did not provide any feedback to the user on their current input, only on 1 input if there was an error after submitting meaning the user would have to redo the entire function and even if they correct the value highlighted in the error, if there were other invalid values that wouldn’t be revealed until after the first error is fixed. There was also no confirmation so accidentally typing the wrong input could cause a lot of work to be required to undo the change. Using the design principles for all features such as this with high visibility, clear affordances, and transference, we will make our interface intuitive to use with minimal or no training and just pre-existing knowledge. Through our design, the heuristics were well followed with the exception of two being flexibility and documentation which could be improved in further iterations. Overall, this redesign significantly improved on many of the shortcomings of the original command line design making for a much better user experience for both beginners and advanced users.

Design Principles

Principle 1 of 20
Visibility:

Designed system using the visibility principle on buttons and menu options. Used to ensure objects were large enough to be perceived and distinguishable from surrounding objects. The original design presented an abundance of unneeded information causing confusion, while hiding important information like what data a function needed, therefore the redesigned project accommodated visibility in the redesign. This increased visibility makes the interface much more transparent to use as thing such as information needed to complete a function are not hidden behind the function call and can be seen clearly on the page.

Familiarity:

The redesigned project uses skeuomorphic symbols and symbols common in other artifacts for similar functions. The original system did not use any principle of familiarity to reduce the cognitive load of available interactions behavior or functionality. It was desired in the redesign to use the principle of familiarity to offload the cognitive tasks of users and help form a mental model for the functionality of the provided interactions. Examples include an eye to display or hide password entry, a calendar icon to display date selection functionality, an Individual icon for updating personal info, magnifying glass for searching.

Affordance:

The original system did not make any use of affordances. Operating the system required reading instructions constantly even for simple functions and the information needed for a function to operate was not known until the function had been called. With the redesign, perceived affordances are used to make navigation simple and function operation obvious. For navigation, hovering over a tab on the left changes the mouse to the clicker icon and the tab highlights inviting the user to click which is how they can change the function tab. In the functions, buttons like submit, calendar, or drop downs, highlight when hovered over indicating they also are to be clicked for the action to occur.

Constraints:

Constraints help minimize errors and improve usability and the use of non-physical constraints improve the clarity of the design. The original system had no constraints on the input in the communication through the input of commands. This led to additional text to clarify the use and errors occurring from typing incorrect formats or invalid input through the text commands. The use of constraints in the re-designed system included date selection from a calendar to constrain the user only to provide a valid format of data to the system, dropdown menus were used for the selection of one option from a list of valid inputs an improvement of being unconstrained and able to type any string of text limiting the potential options the user could select. In addition, with the design decision to apply the constraints principle the clarity of interaction was improved and unneeded information regarding valid inputs and format could be removed.

Mapping:

The mapping principle was utilized in the re-design to have single controls map to a single function. Having this one-to-one relationship of control to function was chosen to increase usability and avoid confusion that is caused by a single control having many functions associated. Every button and data entry control only produces one reaction function, and the use of a meaning mapping of submission buttons is blue following a common convention of submission buttons. For behavior mapping, the control is to input a number into the field uses an increase and decrease arrow up and down which provides a clear mapping of action to reaction that up arrow will increase the inputted number and down arrow decrease the number.

Fitts’ Law:

Fitts’ law was considered in the design of the system and used for the menu selection buttons of the system. Applying this principle for the menu was to put it on the left edge of the screen to create an effect of larger size by the left side of the button having an infinite edge. The effect of this and the large vertical size on the menu options decreases the time to move the cursor over the menu buttons because of size reducing time to move over the button compared to if the button was not on the edge or if the size was smaller.

Consistency:

Our interface was designed to ensure consistency among its interaction groups. All function navigation is done on the left-hand side through clicking with no hidden functionality, all the functions operate by using separate boxes for each data input with consistent input characteristics with text for free data, the calendar for dates, and drop downs for restricted data. This is done so that once a user has used one part of the interface, that knowledge easily transfers to other parts so it is very simple to use and learn.

Feedback:

The original command line interface made limited use of feedback with only success or error messages on function submission without much indication as to why errors occurred. Our redesign makes better use of feedback through providing better detail in both success and error pop ups but also feedback is provided before the function is submitted. Required data points have a red star to indicate they must be filled out and are not optional whereas on the old interface the user just had to know. Feedback is also given by highlighting text as its being input in either red or green. The text is highlighted red if it is invalid such as for employee ID it must be 6 numbers so if it is less than 6 or more than 6 the text is red indicating a problem with the input to the user but if the text is valid by being 6 numbers then it turns green giving feedback to the user that it is a valid input.

Transfer:

The redesigned system makes use of positive conceptual transfer by taking design inspiration from similar systems. The login page operates the same as all login pages, the function selection being on the left is a common feature of other systems such as email inbox selection, the logout button being on the top right is also very common in other systems, required inputs having a red star is also a very common feature. These are just a few but they allow knowledge transfer from using other systems to help the user understand how to navigate and use this system.

Metaphors:

The original design was text only so it did not make use of any metaphors however the re-design makes use of metaphors to aid in navigation and some functions where format can be confusing. The system uses metaphors to help with navigating by having icons that match the user’s mental model of a real world object’s function to the virtual function. Examples include a magnifying glass for search and a calendar for reporting work hours. This enables collapsing the function descriptions and makes it easy to navigate using just the icons alone and also helps the user understand the function’s purpose. Another metaphor is the calendar for date inputs which mimics manipulating a physical calendar. This is useful as date formats can be confusing to users since ordering and length of each value varies across systems and regionally so having this manipulating metaphor allows the user to easily select the date as they would with a physical calendar and that will keep the date format consistent and easy to follow for new users who may be used to a different date format than implemented in the system. Text is also highlighted either green or red depending on if it follows a valid format for submission with green being valid and red being invalid as a metaphor for traffic lights which use the same colours to indicate go or stop.

Control:

The control principle was used in the re-design to enable users with more experience and a better mental model of the system to focus more on the direct task being done over the additional information used in the navigational menu. The menu is collapsable to just icons or expanded to have icons and text. The text helps give newer users easier learning of the system and navigation to functions with slightly less space to focus on the current task and advanced users that are familiar with the position of different tasks to navigate to can have more of available screen space to improve visibility and ease of interaction with the current active task.

Signs:

The system navigation bar makes use of signs for functions which do not have a good metaphor. It makes use of iconic signs such as for adding a new staff member the icon is a person with a plus sign which signs to the user that a person is being added. Some are more abstract indexical signs such as the employee role which is a person on a clipboard indicating something is being changed about a person. Some are abstract signs like team management being a box connected to other boxes which is an abstract representation of the connections in a team and organization. These signs along with the metaphors work together to allow the navigation bar to be collapsed while still giving indication as to what the functions do.

Gestalt Laws of Grouping:

The design uses colour similarity and grouping to separate the interface into two perceptual sections. One section being the navigation, and one section being the function operation. The principle of similarity was used by having all the navigation tabs in the same colour text, same colour background, and same size icons which is a different text and background colour than the function operation which groups them visibly into separate entities. We also used the principle of grouping by putting the tabs close together while the function operation functions are more spread apart which also differentiates them and makes them perceptually appear as separate from the navigation tab.

Fidelity:

A high degree of fidelity was chosen in our design for representing the data in the system and representations used for input data, providing and focusing only on the data relevant to task completion. The design called for a high degree of fidelity as a large amount of information is not needed to be displayed to function so potentially increasing the density by reducing the fidelity was not needed. The redesign chose for including all relevant information cutting out information in the original representation that was not required for the tasks and presenting the relevant information for each task with high fidelity.

Progressive Disclosure:

Progressive Disclosure is used in the system to avoid overloading the user with information. We use progressive disclosure through our system navigation tabs. Instead of having many functions on a single page and overloading the user so they don’t know where to input or where to submit to get the function they want, we use the tab system to only reveal the functionality that the user is looking for.

Learnability:

The system was designed to be very simple to use. Since it was designed for companies which could have many employees which they don’t want to pay hours to have to learn how to use a program, the system takes advantage of users knowledge along with simple clear design so that at a glance it can be clear to the user how the system works making it very easy to learn to be used proficiently without any previous instruction.

Discoverability:

Discoverability in interaction design means that the user must discover what features are included in the UI within a microsecond. In our design, all features are displayed in the left side bar, and all feature icons are accompanied by a text label. This allows users to use our system without any road blocks.

Simplicity:

For our web-based management system, we did our best to make it look simple. This makes it very easy for users to find the features they need through the use of high contrast between backgrounds and important information along with keeping the page well spread out to avoid clutter causing confusion. Of course, we still use shading, colouring, and formatting to make it aesthetically. We successfully get the right balance between aesthetic appeal and the optimal amount of information in each page.

Navigability:

It is quite obvious in our system, since all available features are neatly arranged from top to bottom and listed in the sidebar. When the mouse cursor approaches a specified function, the area around that function deepens in colour. This prompts the user to click on that area. This allows the user to move smoothly through them to reach the option they want.

Findability:

Findability refers to the degree to which a particular object is easy to discover. We use a calendar-like date picker in our project.In this selector, dates that are far away from today can be easily selected. On the contrary, when we select a date that is far from today, we can also easily return to the current date with the “Today” button.

Heuristic Evaluation

  1. Visibility of System Status

    2. With red asterisks to show what information is required, a real-time changing input font colour to indicate whether the user’s input is legitimate, and an informative popup window that appears after submitting, our system keeps the user informed, and provides appropriate feedback at all times. In conclusion, our system has a high visibility of system status

  2. Match Between the System and the Real World

    3. Our system speaks the users’ languages, our system is designed mostly for regular users with no background of professional IT or database knowledge. Since most of the information users get is whether their operation was executed successfully or not, and this is largely determined by whether the database execution returned an error, we carefully designed the user friendly popup messages to map these database returns by containing only words, phrases, and concepts familiar to the users, and eliminating any technical jargons and return codes. When users try to enter illegal inputs, their input font will first turn red, when users try to submit such inputs, the system will print a red message box and block the submit, when users try to submit a legal but meaningless command, the popup window will provide a pop-up window with corresponding information. In conclusion, there’s a good match between the system and the real world, and information always appears in a natural and logical order.

  3. User Control & Freedom

    4. Our system does not provide a direct undo or redo operation. However, each action users do on our system can be reverted or modified by other actions. For example, If users update personal information by mistake, they can always redo/undo that by doing the same action again with the old values. If users reported wrong work hours in a day, the users’ manager can view the changes and correct them. So in some cases, users may not be able to exit an unwanted state without other users with higher authority. As a result, our system provides limited User control and freedom.

  4. Consistency and Standards

    5. The actions in our system are clearly separated in the sidebar by different names and Icons, when users are doing an action, other actions are hidden until they click the tab on the sidebar. One action can only be done in one specific way. So there’s little chance users think two actions mean the same thing, there’s high consistency and standards in our system.

  5. Error Prevention

    6. The system makes use of error prevention in multiple ways. For complex inputs such as dates, we use a calendar selector which will fill in the date for the user which helps with confusing the many different date formats. Another way it performs error prevention is through text highlighting. Values highlight in red if they are not valid such as if it must be 8 characters and is less or more, it will highlight in red but if it is 8 valid characters it will highlight in green. This helps the user know that their input is valid or invalid before they have submitted it. If they still submit invalid inputs, the value is highlighted with a simple error message preventing it from occurring again.

  6. Recognition vs. Recall in User Interfaces

    7. The system minimizes the need for the user to recall information by keeping all functionality visible and not hiding or surprising the user with inputs. The old system the user would not know the inputs needed for a function until they were to be entered which meant they either had to remember or they had to constantly go between the system and their info however the redesign keeps all inputs in the open which allows the user to know what inputs are needed as soon as they are on the page. It also has all functionality visible on the side navigation bar so no need to remember commands to call functions and no functions where are hidden behind multiple menus to get to so there is low cognitive load with very little to remember about the system in order to use it effectively.

  7. Flexibility and Efficiency of Use

    8. The system has some flexibility for experienced users in the form of the collapsable navigation bar which keeps only the icons so experienced users will still know what function to select while giving more screen real estate to the actual function inputs and results. However there is not much flexibility besides that such as there are not any custom colour schemes for colour blind users and such.

  8. Aesthetic and Minimalist Design

    9. In our system, users get information from the sidebar and content texts, font colour, red text box blocking submits and popup windows. All text areas contain sufficient and brief information designed to be user-friendly. Our system presents little to no extra information that users do not need, so our system is highly aesthetic and minimalist designed.

  9. Help Users Recognize, Diagnose and Recover from Errors

    10. The system returns error messages which properly highlight what caused the error to occur which helps the user correct the values when they re-do the function. For inputs which are length limited or must be numbers only, if the input does not match this it will give a message which explains the issue with their input so they can correct it. If the values are valid but there is another issue with submission such as trying to set an inactive employee to inactive it provides an explanation stating why the function did not run properly so the user can decide how to proceed from there.

  10. Help & Documentation

    11. The system was designed to be very simple to use and as such we did not include any documentation. It should be self-explanatory how to use the system but a simple help tab with some basic instructions could have been helpful to improve the learning process.

Improvement Recommendations

In light of our heuristic evaluation, there are some improvements which can be done to the system. The main two heuristics which fell short were Flexibility and Efficiency of Use and Help and Documentation. For Flexibility and Efficiency of Use our interface was not very customizable. Two simple improvements that can be done would be regarding the colour scheme being fixed and navigation bar ordering being fixed. For the colour scheme, we could allow the user to customize their colouring with presets for different types of colour blindness which would make the interface more accessible to colour-blind people if their colour blindness causes the contrast in the current default colour scheme to be low making it difficult to use. We can also allow the user to reorder the navigation bar so that they can group functions as they like or move commonly used functions closer to the top, based on their use case. This can make it so that finding and selecting functions is more efficient as it will be to their preferences. The other heuristic of help and documentation could be improved by having a navigation tab for help where a small wiki could be written, explaining basics about operating the system. Question marks near inputs could also be helpful for explaining input value meaning and expected format such as saying 4 number code or alphanumeric up to 20 characters. This could make the system easier to learn for new users and also be nice for users who forget or are confused why the input is highlighting red when they expect it to be valid. In addition to that, we need to consider increasing the depth of the page, where all our current functions are implemented in one page. For some of the functions, a large amount of input is required or a large amount of data is returned from the database, which makes the page look very crowded. For future improvements, we can consider adding a scroll bar or adding the ability to switch pages to make the overall page more concise.
